# Strict Retrieval Validation

This document describes the new strict retrieval validation flow for full-tutorial retrieval.

## Why A New Strict Dataset Is Needed

The legacy retrieval dataset labels only relative file paths such as `system/controlDict`.
That is acceptable when retrieval is scoped to a single case, but it breaks down once we
retrieve across all tutorials because many cases share the same relative paths.

The audit generated by `build_strict_retrieval_dataset.py` shows:

- `120` legacy retrieval entries in total
- `6` entries can be uniquely resolved to one case
- `108` entries are ambiguous across multiple cases
- `6` entries point to files that do not exist in the current tutorial corpus

Because of that, the legacy dataset should not be used for strict `case + file` evaluation.

## New Strict Dataset

The strict dataset is generated at:

- `dataset/retrieval/blastfoam_retrieval_validation_dataset_strict.json`

It is built from the case-specific modification datasets and uses the canonical retrieval unit:

```text
case_path::file_path
```

Each entry includes:

- `query`: the natural-language modification request
- `description`: the scenario-level description
- `modification`: the concrete change specification
- `case_path`: the resolved tutorial case path
- `target_files`: a list of strict targets with `case_path`, `file_path`, and `canonical_id`
- `difficulty`, `category`, `source_dataset`, `case_name`

## Validation And Auditing

The builder also generates:

- `dataset/retrieval/tutorial_case_manifest.json`
- `dataset/retrieval/blastfoam_retrieval_validation_dataset_legacy_case_audit.json`
- `dataset/retrieval/blastfoam_retrieval_validation_dataset_strict_audit.json`
- `dataset/retrieval/STRICT_RETRIEVAL_AUDIT_SUMMARY.md`

Current strict-audit status:

- `122` pass
- `8` warn
- `0` fail

The remaining warnings are heuristic review prompts from identifier matching.
They do not indicate missing files or broken case resolution.

## Strict Evaluator

The evaluator lives at:

- `dataset/retrieval/strict_retrieval_evaluator.py`

It expects retrieval results that can be normalized into strict targets. Supported forms include:

- `case_path::file_path`
- `{ "case_path": "...", "file_path": "..." }`
- absolute tutorial file paths
- relative full paths such as `blastFoam/freeField/system/controlDict`

The evaluator computes:

- `hit@k` based on exact `case_path::file_path` matches
- `case_hit@k` based on case-only matches
- `file_hit@k` based on file-only matches
- `precision@k`
- `recall@k`
- `mrr`

## Build Command

```bash
python dataset/retrieval/build_strict_retrieval_dataset.py \
  --tutorials-dir /path/to/blastFoam_tutorials
```
