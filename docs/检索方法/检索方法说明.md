# 基于算例内容知识图谱的检索方法技术说明

## 1. 概述

本文档详细阐述在 PrincipiaBlastFoam 项目中，如何利用“算例内容知识图谱”（Case Content Knowledge Graph）进行高效、精准的仿真案例文件和参数检索。该方法的核心思想是将一个完整的 OpenFOAM/BlastFoam 算例（包含其所有物理设置、数值方案和控制参数）转化为一个结构化的图数据，通常表现为 JSON 格式。

与传统的基于文本或向量的检索不同，这种知识图谱能够捕捉文件、参数和算例之间的层级和定义关系，为大型语言模型（LLM）提供深度语义理解能力。它使得 AI 助手能够：

-   **快速理解**：在短时间内掌握一个复杂算例的完整结构和核心参数。
-   **精确定位**：当用户提出修改需求时（例如“修改模拟结束时间”），能准确找到对应的文件和参数位置。
-   **自动化操作**：为自动化修改、比较和生成新算例奠定坚实的数据基础。

## 2. 知识图谱构建

知识图谱的构建过程是将一个算例目录下的所有相关文件解析为图的节点和关系。

### 2.1. 数据源

知识图谱的数据完全来源于 OpenFOAM/BlastFoam 的算例文件本身。每一个算例目录都被视为一个独立的知识图谱来源。

### 2.2. 图谱模式（Schema）

为了有效地组织知识，我们定义了以下核心节点（Node）和关系（Relationship），其结构专为图数据库（如 Neo4j）设计。

-   **核心节点**:
    -   `Case`: 代表整个算例的根节点。
        -   *属性*: `name` (算例名), `path` (相对路径)。
    -   `File`: 代表算例中的一个具体文件。
        -   *属性*: `name` (文件名), `path` (文件相对路径)。
    -   `Variable`: 代表文件中的一个变量、关键词或参数。
        -   *属性*: `name` (变量名), `type` (变量的具体分类，如 `Parameter`, `BoundaryCondition` 等)。

-   **核心关系**:
    -   `CONTAINS`: 表示一个算例（Case）包含一个文件（File）。
        -   *示例*: `(Case)-[:CONTAINS]->(File)`
    -   `DEFINES`: 表示一个文件（File）定义或使用了一个变量（Variable）。
        -   *示例*: `(File)-[:DEFINES]->(Variable)`

通过这种方式，我们将一个算例的目录结构和文件内容，转换为了一个描述其内在逻辑的互联网络。

## 3. 检索流程

基于算例内容知识图谱的检索流程，旨在将用户的自然语言需求转换为结构化的图查询，从而精准地定位到目标文件或参数。

1.  **自然语言理解 (NLU)**:
    -   使用大型语言模型（LLM）对用户的输入（例如 "将爆炸物的位置移动到 (1 2 3)"）进行意图识别和实体提取。
    -   从查询中识别出核心实体，如 `Variable.name: highExplosiveChargePosition`，以及期望修改的值 `(1 2 3)`。

2.  **Cypher 查询生成**:
    -   根据提取出的实体和意图，动态生成一个或多个 Cypher 查询语句。
    -   例如，为了找到定义了 `highExplosiveChargePosition` 参数的文件，可以生成如下查询：
        ```cypher
        MATCH (f:File)-[:DEFINES]->(v:Variable {name: 'highExplosiveChargePosition'})
        RETURN f.path
        ```

3.  **图数据库查询**:
    -   使用 `neo4j` Python 驱动程序连接到图数据库。
    -   执行生成的 Cypher 查询，获取目标文件的路径。

4.  **文件操作与返回**:
    -   根据查询返回的文件路径，AI 代理可以读取该文件。
    -   在文件内容中定位到 `highExplosiveChargePosition` 参数，并将其值修改为 `(1 2 3)`。
    -   将修改后的结果或状态返回给用户。

## 4. 优缺点分析

### 4.1. 优点

-   **定位精准**: 能够直接、准确地找到定义特定参数的文件，避免了模糊的文本搜索。
-   **结构化理解**: AI 不再将算例视为一堆无组织的文本文件，而是理解其内在的结构和逻辑。
-   **可解释性强**: 检索路径（`Case -> File -> Variable`）清晰可见，易于理解和调试。
-   **自动化潜力巨大**: 为跨案例比较、参数一致性检查和自动化案例生成等高级功能提供了可能。

### 4.2. 缺点

-   **构建成本**: 需要为每个算例预先生成知识图谱 JSON 文件，并将其导入图数据库，存在一定的预处理成本。
-   **查询转换**: 将复杂的自然语言准确地转换为 Cypher 查询仍存在挑战，依赖于 LLM 的能力。
-   **维护同步**: 当算例文件发生变化时，需要有机制来更新或重新生成知识图谱，以保证数据的一致性。

## 5. 结论

基于算例内容知识图谱的检索方法，为 PrincipiaBlastFoam 项目提供了一个强大、精准且可解释的知识引擎。它将用户需求与底层代码文件之间建立了可靠的桥梁，是实现 OpenFOAM/BlastFoam 算例高度自动化和智能化的关键技术。
