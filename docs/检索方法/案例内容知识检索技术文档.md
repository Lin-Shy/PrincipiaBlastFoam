# 案例内容知识检索技术文档

## 1. 定位

`CaseContentKnowledgeGraphRetriever` 是当前项目中用于 tutorial 案例内容检索的核心方法说明。

它服务的不是“单个 case 文件夹内检索”，而是“在所有 tutorial 中先找对 case，再找对 file”的全局检索场景。当前评测口径也已经切换为 strict 的 `case_path::file_path`。

相较于纯关键词或纯向量检索，这个方法的核心优势是：

- 先利用知识图谱中的结构关系理解案例、文件、变量之间的联系
- 再由 LLM 动态决定检索路径，而不是一次性硬编码规则
- 必要时回读 tutorial 原始文件内容，避免只靠图谱摘要做判断

## 2. 数据来源与运行方式

### 2.1 数据来源

案例内容知识图谱来自：

- `data/knowledge_graph/case_content_knowledge_graph/`

目录下每个 JSON 文件对应一个 tutorial case 的知识图谱，内容包含：

- `Case` 节点
- `File` 节点
- `Variable` 节点
- `CONTAINS` / `DEFINES` 等关系

### 2.2 运行方式

当前实现并不依赖 Neo4j 或 Cypher 查询。

实际代码会把多个 JSON 图谱文件加载到内存中，并在本地完成：

- 节点与关系索引构建
- case 到 file 的映射
- file 到 variable 的映射
- 基于 LLM 生成策略的图内搜索与重排

对应实现位于：

- `principia_ai/tools/case_content_knowledge_graph_tool.py`

## 3. 知识表示

### 3.1 节点类型

- `Case`: 一个完整教程案例
- `File`: 案例中的具体文件
- `Variable`: 文件中的参数、关键词或模型项

### 3.2 关键关系

- `Case -[CONTAINS]-> File`
- `File -[DEFINES]-> Variable`

这种表示使检索器不仅能回答“哪个文件里有这个词”，还能回答“这个需求通常应该落到哪个案例、哪个配置文件族”。

## 4. 检索流程

### 4.1 图谱加载与命名空间隔离

启动时，检索器会加载 `case_content_knowledge_graph` 目录下所有 JSON 图谱，并为节点 ID 添加来源文件前缀，避免不同案例之间出现 ID 冲突。

### 4.2 ReAct 决策循环

检索器以 ReAct 方式运行：

1. 根据用户查询和图谱摘要让 LLM 生成下一步动作。
2. 在本地图谱中执行动作。
3. 把观察结果写回 history。
4. 继续迭代，直到结束或达到最大轮数。

LLM 可选动作只有三类：

- `search_nodes`: 按标签、属性、关键词组合搜索节点
- `inspect_nodes`: 查看节点详情，并在需要时读取真实文件内容
- `finish`: 输出最终结果

### 4.3 本地搜索与重排

`search_nodes` 并不是直接调用数据库查询，而是由本地搜索逻辑执行：

- label 精确匹配
- property 过滤
- keyword 模糊匹配
- 基于 case 线索与 file 线索的补充分数

在得到候选节点后，检索器会进一步把节点结果解析成结构化的 case/file 结果，并尽量把结果收敛到更一致的同一案例上下文中。

### 4.4 文件内容增强

如果设置了 `BLASTFOAM_TUTORIALS`，`inspect_nodes` 可以回读 tutorial 中的真实文件内容，用于：

- 校验目标文件是否合理
- 补充图谱中没有完整展开的细节
- 为智能体后续编辑提供更贴近原文件的上下文

## 5. 对外接口

### 5.1 `search`

返回文本结果，适合直接给智能体阅读。

### 5.2 `search_detailed`

返回结构化结果，适合评测与程序化消费。当前会输出：

- `text`: 文本说明
- `structured_results`: 结构化命中列表
- `node_ids`: 最终使用到的图节点

`structured_results` 中的每条结果至少包含：

- `case_path`
- `file_path`
- `canonical_id`
- `path`
- `score`

这也是 strict 评测脚本使用的接口。

## 6. Retrieval LLM 配置

案例内容检索现在支持独立的 retrieval LLM 配置，不再必须与主工作流智能体共用同一组 LLM 参数。

配置优先级如下：

1. 初始化 `CaseContentKnowledgeGraphRetriever(...)` 时显式传入
2. `.env` 中的 `RETRIEVAL_LLM_API_KEY` / `RETRIEVAL_LLM_API_BASE_URL` / `RETRIEVAL_LLM_MODEL`
3. `.env` 中的 `LLM_API_KEY` / `LLM_API_BASE_URL` / `LLM_MODEL`

对应辅助逻辑位于：

- `principia_ai/tools/retrieval_llm_config.py`

## 7. 优势与边界

### 7.1 优势

- 能显式区分 case 与 file，不再只看局部文件路径
- 对隐式目标文件、目标导向查询更友好
- 检索路径可解释，便于调试失败案例
- 可以和真实 tutorial 文件内容联动

### 7.2 边界

- 仍依赖 LLM 生成搜索策略，速度通常慢于 embedding 基线
- 对特别隐式的编辑定位问题，仍可能需要更多案例先验
- 图谱覆盖范围决定了可检索上限，如果 tutorial 更新而图谱未同步，效果会下降
