# 案例内容知识检索技术文档

## 1. 概述
`CaseContentKnowledgeGraphRetriever` 是本项目中用于从 BlastFoam 教程案例（Tutorial Cases）中提取信息的智能工具。与传统的关键词搜索不同，该工具采用了 **ReAct (Reasoning + Acting)** 代理模式，能够像人类专家一样，通过“思考-行动-观察”的循环，在复杂的案例文件结构中主动探索答案。

该工具结合了**知识图谱的结构化查询能力**和**文件系统的实时读取能力**，既能理解案例间的关联，又能获取最新的代码细节。

## 2. 核心架构

### 2.1 知识图谱模型
该工具基于从 OpenFOAM/BlastFoam 案例文件中提取的知识图谱，主要包含以下实体关系：
- **Case (案例)**: 代表一个完整的仿真案例（如 `blastFoam_building3D`）。
- **File (文件)**: 案例中的配置文件（如 `system/controlDict`）。
- **Variable (变量)**: 文件中定义的参数（如 `endTime`, `p`）。
- **关系**:
    - `(Case) -[CONTAINS]-> (File)`
    - `(File) -[DEFINES]-> (Variable)`

### 2.2 ReAct 代理机制
工具实现了一个自定义的 ReAct 循环，允许 LLM 动态决定检索策略，而不是执行固定的查询逻辑。

#### 循环流程 (`search` 方法)
1.  **初始化**: 加载知识图谱，生成图谱结构摘要。
2.  **循环执行 (Max Iterations)**:
    - **观察 (Observation)**: 获取当前知识图谱的统计摘要 (`_get_knowledge_graph_summary`) 和历史操作结果（History）。
    - **思考 (Thought)**: LLM 分析用户意图和当前掌握的信息，决定下一步行动。
    - **行动 (Action)**: 执行具体工具（搜索节点、查看详情、结束）。
    - **记录**: 将行动结果追加到 History 中，供下一轮思考使用。

### 2.3 可用工具 (Tools)

代理可以调用以下三种内部工具：

#### 1. 节点搜索 (`search_nodes`)
- **功能**: 在知识图谱中查找符合条件的节点 ID。
- **实现**: `_execute_search_strategy`
- **支持过滤**:
    - `node_labels`: 节点类型 (如 "Case", "File", "Variable")。
    - `property_filters`: 属性精确匹配 (如 `name="controlDict"`).
    - `keyword_search`: 关键词模糊匹配，用于对结果进行打分排序。
- **应用场景**: 当不知道具体 ID，只知道名字或概念时使用（例如：“查找所有使用了 k-epsilon 模型的案例”）。

#### 2. 节点探查 (`inspect_nodes`)
- **功能**: 查看特定节点的详细信息。
- **实现**: `_retrieve_node_details_with_content`
- **增强功能**:
    - **文件内容读取**: 如果节点是文件或包含文件路径，工具会调用 `_get_file_content` 读取磁盘上的实际文件内容（支持 `BLASTFOAM_TUTORIALS` 环境变量）。
    - **上下文补充**: 自动调用 `_get_case_info_for_nodes` 查找节点所属的 Case 信息，确保返回的代码片段有明确的案例背景。
- **应用场景**: 当已经找到目标文件或变量，需要查看具体代码设置时使用。

#### 3. 结束任务 (`finish`)
- **功能**: 生成最终回复并退出循环。
- **输入**: 最终确定的相关节点 ID 列表和解释。

## 3. 代码实现细节

### 类结构 (`CaseContentKnowledgeGraphRetriever`)
- **`_load_knowledge_base`**: 
    - 从 `data/knowledge_graph/case_content_knowledge_graph` 目录加载分散的 JSON 图谱文件。
    - 构建内存中的节点列表和关系列表。
- **`_react_decide`**: 
    - 核心决策函数。
    - 构建包含 System Prompt、KG Summary、User Query 和 History 的 Prompt。
    - 要求 LLM 输出 JSON 格式的 Action。
- **`_get_file_content`**:
    - 连接物理文件系统的桥梁。
    - 读取实际的 OpenFOAM 字典文件，提供比图谱元数据更准确的配置信息。

### 关键技术点
- **混合检索 (Hybrid Retrieval)**: 
    - **元数据层**: 使用知识图谱快速定位文件位置和关系。
    - **内容层**: 使用文件读取获取最新的配置细节。
- **动态策略**: LLM 不是一次性生成查询，而是根据中间结果调整搜索方向。
    - *例子*: 用户问“building3D 案例的结束时间是多少？” -> 代理先搜 "building3D" 找到案例 ID -> 再搜该案例下的 "controlDict" -> 最后查看 controlDict 内容找到 endTime。
- **容错处理**: 包含对 LLM 输出格式错误（如 JSON 解析失败）和文件读取失败（如路径不存在）的异常处理机制。

## 4. 环境变量依赖
- `BLASTFOAM_TUTORIALS`: 指向 BlastFoam 教程目录的本地绝对路径。工具利用此路径读取实际的字典文件内容。如果未设置，只能返回图谱中的元数据。
- `LLM_API_KEY` / `LLM_API_BASE_URL`: 模型服务配置。
