# 用户手册知识检索技术文档

## 1. 概述
`UserGuideKnowledgeGraphRetriever` 是本项目中用于从 BlastFoam 用户手册中检索信息的专用工具。当前实现采用“分层召回 + 图谱感知重排”的策略：

- 先用 LLM 在章节、小节、子小节层面逐步缩小范围。
- 再基于完整知识图谱中的节点文本、层级和祖先路径做确定性重排。
- 最后返回结构化 top-k 结果和对应正文内容。

这样做的原因是，纯分层 LLM 筛选虽然能快速命中正确章节，但容易把答案停留在父级 `sec*` 节点，无法稳定定位到真正需要的 `model*`、`method*`、`solver*`、`utility*` 等细粒度节点。图谱感知重排正是为了解决这个问题。

## 2. 核心架构

### 2.1 双知识库设计
为了兼顾检索效率和内容完整性，工具同时使用两个版本的用户手册知识库：

1. `simplified_user_guide_knowledge_graph.json`
   - 内容: 章节号、标题、层级关系、摘要
   - 用途: 用于 chapter / section / subsection 三阶段 LLM 召回
2. `user_guide_knowledge_graph.json`
   - 内容: 完整节点正文、摘要、类型、父子关系
   - 用途: 用于候选重排和最终内容返回

### 2.2 图谱索引
加载完整知识库后，检索器会预计算一套轻量索引，供重排使用。当前会利用以下字段：

- `id`
- `number`
- `title`
- `semantic_type`
- `content_summary`
- `content`
- 祖先章节 / 小节标题
- 父子关系、节点深度、是否为叶子节点

这些信息会被转成标准化文本和 token 统计，用来给 query-node 对打分。

## 3. 检索流程

### 第一步：章节识别
- 方法: `_identify_relevant_chapters`
- 输入: 用户 query
- 逻辑:
  - 从简化图谱中取出所有 `ch*` 节点
  - 将章节号和标题交给 LLM 判断相关范围
- 输出: 候选 chapter ID 列表

### 第二步：小节识别
- 方法: `_identify_relevant_sections`
- 输入: 用户 query + 候选 chapter
- 逻辑:
  - 在选中 chapter 下取出一级 section
  - 将 section 标题和摘要交给 LLM 二次筛选
- 输出: 候选 section ID 列表

### 第三步：子小节识别
- 方法: `_identify_relevant_subsections`
- 输入: 用户 query + 候选 section
- 逻辑:
  - 递归收集 section 下的全部子孙节点
  - 同时保留父 section 本身
  - 让 LLM 从完整子树中找更精细的 subsection / model / method 编号
- 输出: 候选 subsection 或 detail 节点 ID 列表

### 第四步：图谱感知重排
- 方法: `_rank_candidates`
- 输入: 用户 query + 前三步得到的范围提示
- 逻辑:
  - 在完整图谱节点上计算 query-node 分数
  - 结合标题命中、短语命中、摘要命中、祖先路径命中、编号命中
  - 根据 query 语义推断目标粒度:
    - `chapter`
    - `section`
    - `detail`
  - 对 section 类问题提高 `sec*` 节点权重
  - 对 detail 类问题提高叶子 `model* / method* / solver* / utility*` 节点权重
  - 将前三步的 LLM 输出作为 scope bonus，而不是最终硬裁剪条件
- 输出: 最终排序后的 top-k 节点

### 第五步：正文提取
- 方法: `_retrieve_full_content`
- 输入: 重排后的节点 ID
- 逻辑:
  - 从完整图谱中提取节点正文、摘要、表格等内容
  - 组织成 agent 和人工都易读的格式化文本

## 4. 代码实现要点

### 类结构
- `__init__`
  - 初始化 retrieval LLM
  - 加载简化图谱和完整图谱
  - 构建 `id_to_node`、`number_to_node_ids`、`parent_to_children` 等索引
  - 预计算重排用的文本与层级特征
- `search_detailed`
  - 统一串联“分层召回 -> 重排 -> 内容提取”
  - 返回 `structured_results`，供 benchmark 评测脚本直接使用
- `search`
  - 对 `search_detailed` 的文本包装接口
  - 供 agent 工作流直接调用

### 关键技术点
- ID 归一化: 兼容 `Section 5.2`、`5.2.3`、`sec5.2.3` 等不同引用形式
- 递归子树展开: 避免遗漏深层模型节点
- 祖先路径注入: 让 `WenYu`、`RK4`、`blastEulerFoam` 等节点在脱离父节标题时仍然可排序
- 粒度感知: 避免 “Which section ...” 和 “Which model ...” 共用一套排序偏好
- 结构化结果输出: 返回 `canonical_id / node_id / section_id / chapter_id / score`

## 5. 2026-04-07 评测快照

以下结果均来自 `2026-04-07` 在 `graph-py310` 环境中的实际运行：

- 数据集:
  `dataset/retrieval/benchmarks/user_guide/user_guide_retrieval_validation_dataset.json`
- 旧 KG 结果:
  `experiments/retrieval_method/results/user_guide/knowledge_graph_retrieval_20260407_152215.json`
- 改进后 KG 结果:
  `experiments/retrieval_method/results/user_guide/knowledge_graph_retrieval_20260407_154114.json`
- embedding baseline:
  `experiments/retrieval_method/results/user_guide/embedding_retrieval_node_20260407_151152.json`

聚合指标如下：

| 方法 | MRR | Hit@1 | Hit@3 | Hit@5 |
| --- | ---: | ---: | ---: | ---: |
| 旧版 KG | 0.5053 | 0.5000 | 0.5106 | 0.5106 |
| 改进后 KG | 0.9610 | 0.9255 | 1.0000 | 1.0000 |
| Embedding baseline | 0.8754 | 0.7872 | 0.9574 | 1.0000 |

这次改动的主要收益是把大量原来只能命中父级 `sec*` 的查询，推进到了真实目标节点。例如：

- `WenYu`、`GidaspowErgunWenYu`
- `LSZK`、`AbelNobel`、`BKW`
- `RK4`、`SSP Runge-Kutta`
- `blastFoam`、`blastEulerFoam`、`blastParcelFoam`
