# 毕业设计中期进展汇报说明

## 1. 课题目标

本项目的目标是构建一个面向 `OpenFOAM / blastFoam` 仿真任务的智能辅助系统，使其能够从用户自然语言需求出发，自动完成知识检索、案例配置、仿真执行与结果分析等环节。当前实现以 `PrincipiaBlastFoam` 为核心，采用基于 ReAct 的多智能体协作框架，并引入知识图谱增强检索来提升专业知识调用的准确性。

从当前仓库状态来看，中期阶段已经形成了三条较为明确的成果主线：

1. 已完成多智能体系统框架搭建，具备任务规划、案例设置、执行与审查等核心角色。
2. 已构建案例内容知识图谱与用户手册知识图谱，并建立可复现的检索评测基线。
3. 已实现端到端案例质量评估脚本，形成了后续大规模实验验证的统一框架。

## 2. 当前已完成的主要工作

### 2.1 系统架构搭建

项目已经实现如下核心模块：

- `OrchestratorAgent`：负责整体任务规划与调度。
- `PhysicsAnalystAgent`：负责物理需求分析和方案制定。
- `CaseSetupAgent`：负责修改 `0/`、`constant/`、`system/` 等关键案例文件。
- `ExecutionAgent`：负责生成和执行仿真运行流程。
- `PostProcessingAgent`：负责结果提取与可视化。
- `ReviewerAgent`：负责质量检查与输出校验。

中期阶段已经形成了较清晰的对比实验框架：

- Knowledge Graph retrieval：基于图结构、ReAct 决策循环和图谱感知重排。

这意味着当前不仅有方法实现，还有统一 benchmark、统一指标和统一结果输出格式，具备较好的可复现性。方法说明可结合图 `[02_retrieval_pipelines.png](/media/dev/vdb1/linshihao/LLM/PrincipiaBlastFoam/docs/midterm_progress_20260407/figures/02_retrieval_pipelines.png)` 展开。

## 3. 阶段性实验结果

## 3.1 案例内容检索结果

在 `case_content` benchmark 上，Knowledge Graph 方法相较于 Embedding baseline 表现出非常明显的优势：

- Embedding：`MRR = 0.0349`，`Hit@5 = 0.0524`，平均检索时间 `0.40 s/query`
- Knowledge Graph：`MRR = 0.6963`，`Hit@5 = 0.7524`，平均检索时间 `13.69 s/query`

这说明：

1. 图谱法显著提升了“找对目标文件”的能力。
2. 代价是推理式检索耗时明显增加，存在精度与效率的权衡。

推荐展示图：

- `[04_case_content_overall_metrics.png](/media/dev/vdb1/linshihao/LLM/PrincipiaBlastFoam/docs/midterm_progress_20260407/figures/04_case_content_overall_metrics.png)`
- `[05_case_content_granularity_gap.png](/media/dev/vdb1/linshihao/LLM/PrincipiaBlastFoam/docs/midterm_progress_20260407/figures/05_case_content_granularity_gap.png)`
- `[06_case_content_difficulty_breakdown.png](/media/dev/vdb1/linshihao/LLM/PrincipiaBlastFoam/docs/midterm_progress_20260407/figures/06_case_content_difficulty_breakdown.png)`

其中，`05_case_content_granularity_gap.png` 很适合现场重点解释。因为它能直观说明：

- Embedding 在 `case_hit@5 = 67.14%` 时，`exact file hit@5` 只有 `5.24%`；
- 说明向量检索往往能找到“相关案例”，却难以进一步锁定“正确文件”；
- 图谱法通过 Case-File-Variable 之间的显式关系建模，把 `exact file hit@5` 提升到了 `75.24%`。

这个结论非常适合用来论证“为什么这个课题不能只用通用 RAG 或普通 embedding”。

## 3.2 用户手册检索结果

在 `user_guide` benchmark 上，两类方法都取得了较高表现，但图谱法仍进一步提升了精细节点定位能力：

- Embedding：`MRR = 0.8754`，`Hit@1 = 0.7872`，平均检索时间 `0.38 s/query`
- Knowledge Graph：`MRR = 0.9610`，`Hit@1 = 0.9255`，平均检索时间 `1.95 s/query`

这说明用户手册场景本身比案例文件定位更规整，因此 embedding baseline 已有较强效果；但图谱感知重排仍然能把答案从“章节级命中”推进到“目标节点级命中”。

推荐展示图：

- `[07_user_guide_overall_metrics.png](/media/dev/vdb1/linshihao/LLM/PrincipiaBlastFoam/docs/midterm_progress_20260407/figures/07_user_guide_overall_metrics.png)`
- `[08_user_guide_granularity_and_difficulty.png](/media/dev/vdb1/linshihao/LLM/PrincipiaBlastFoam/docs/midterm_progress_20260407/figures/08_user_guide_granularity_and_difficulty.png)`

这里比较值得强调的一个现象是：

- Embedding 的 `chapter_hit@1` 已经达到 `97.87%`，说明它大体上能找到正确章节；
- 但 `exact node hit@1` 只有 `78.72%`，说明细粒度排序仍不稳定；
- 图谱法把 `exact node hit@1` 提升到了 `92.55%`，说明其在精细定位和重排方面更有效。

## 3.3 精度与效率权衡

综合两组 benchmark，可以得到当前阶段很重要的研究结论：

- 对于“需要精确定位目标文件 / 目标节点”的任务，Knowledge Graph 检索显著优于 Embedding baseline。
- 对于用户手册这类结构更清晰、层级更明显的文本，Embedding 已经能完成粗定位，但图谱法仍能提升最终排序质量。
- 图谱法的代价是推理式搜索耗时更长，因此后续工作需要进一步优化检索效率。

## 4. 中期阶段可以如何概括成果

如果需要用一段简短的话概括目前进展，可以采用下面这段表述：

> 当前已完成多智能体仿真辅助系统的总体框架搭建，构建了案例内容与用户手册两类知识图谱，并建立了严格检索评测基线。实验结果表明，知识图谱增强检索在案例文件定位和用户手册细粒度节点定位任务上均优于 embedding baseline，其中在案例内容检索任务上提升尤为显著。与此同时，项目已实现端到端案例质量评估框架，为后续批量案例生成与自动验证实验奠定了基础。

## 5. 当前存在的问题

为了让中期汇报更稳妥，建议主动说明下面几个客观问题：

1. 当前最充分的量化实验主要集中在检索模块，端到端自动生成案例的批量验证结果还需要进一步补齐。
2. 图谱法当前精度高，但推理式检索的耗时仍偏大，后续需要在效率与精度之间做更好的平衡。
3. 知识图谱覆盖范围与教程同步性会影响上限，因此后续仍需完善自动更新与增量维护机制。

这种表述既真实，也能体现你对系统局限性的清楚认识。

## 6. 下一阶段计划建议

答辩中建议把下一阶段计划落到三件事上：

1. 补齐端到端案例生成与验证实验，形成“检索能力提升是否能转化为案例生成质量提升”的完整证据链。
2. 优化 Knowledge Graph 检索效率，例如减少无效推理轮次、引入轻量级预筛选机制。
3. 完善论文写作中的方法章节、实验章节与消融分析，形成更完整的毕业论文结构。
